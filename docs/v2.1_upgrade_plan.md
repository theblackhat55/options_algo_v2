# Go-Live Plan v2.1

## Purpose
This document updates the original phased plan to reflect the current system state and define the **remaining work required** to move from a functioning paper-live pipeline to controlled strict-live and, eventually, small-capital live deployment.

It assumes the following are already in place:

- live Databento daily bars
- live Polygon options chain snapshots
- real ADX / SMA / ATR feature computation
- real `iv_hv_ratio`
- persisted IV proxy history for first-pass `iv_rank`
- watchlist-driven scan flow
- scan explainability / debug artifacts
- paper-live logging and review tooling
- strict-live blocking when placeholder IV inputs remain

---

## Current State Summary

### Completed foundation
- watchlist-driven nightly scan flow exists
- underlying feature pipeline is working in live mode
- options chain normalization and spread selection are working
- paper-live logging/review loop exists
- `iv_hv_ratio` is live and symbol-specific
- `iv_rank` infrastructure exists but requires history accumulation

### Remaining blockers to true strict-live readiness
1. `iv_rank` is not yet fully real across symbols
2. quote-quality / synthetic-quote diagnostics are incomplete
3. broader production hardening controls are incomplete
4. paper execution / fill-quality simulation is incomplete
5. strict-live readiness criteria need to be formalized and enforced consistently

---

# Phase 1 — Complete Real IV Rank

## Goal
Finish the transition from placeholder `iv_rank` to a live, history-based rank for all actively scanned symbols.

## Why this phase comes first
Strict-live is currently blocked primarily because `iv_rank` still falls back to placeholder values when insufficient history exists.

## Tasks
1. Continue daily IV proxy history accumulation for all paper-live scanned symbols
2. Verify per-run metadata:
   - `iv_rank_ready_symbols`
   - `iv_rank_insufficient_history_symbols`
   - `iv_rank_history_path`
   - `iv_rank_trailing_observations`
3. Confirm `iv_rank` stops falling back once trailing observation threshold is met
4. Add a lightweight audit script or report to summarize:
   - symbols ready for real `iv_rank`
   - symbols still on placeholder `iv_rank`
   - observation counts by symbol
5. Validate that `iv_state` behavior changes sensibly once real `iv_rank` becomes available

## Exit Criteria
- all actively scanned symbols have sufficient IV proxy history
- `used_placeholder_iv_rank_inputs == False` for the target paper-live watchlist
- strict-live is no longer blocked by placeholder IV rank

---

# Phase 2 — Quote-Quality and Options Viability Hardening

## Goal
Complete the missing diagnostics around option quote quality and ensure optionability checks are explicit and measurable.

## Tasks
1. Add quote-quality counters to scan/runtime metadata:
   - total quotes fetched
   - quotes with real `last_quote`
   - synthetic quote count
   - missing greeks count
   - zero-delta count
   - zero-open-interest count
2. Add symbol-level options viability diagnostics:
   - contracts near target DTE
   - contracts with acceptable spread width
   - contracts with real quotes
   - contracts with usable greeks
3. Surface these diagnostics in:
   - scan artifacts
   - paper-live symbol logs
   - review scripts where appropriate
4. Define explicit strict-live quote-quality thresholds
5. Block strict-live if option quality falls below policy thresholds

## Exit Criteria
- quote-quality metrics are visible for every scanned symbol
- strict-live can reject poor-quality option chains deterministically
- options viability is measurable and explainable in artifacts/logs

---

# Phase 3 — Broader Universe Operationalization

## Goal
Move from a small filtered watchlist operating loop to a repeatable broader-universe daily prefilter.

## Tasks
1. Finalize the broader universe source:
   - S&P 500
   - major ETFs
   - other liquid optionable names
2. Standardize the daily watchlist build pipeline:
   - underlying prefilter
   - options viability filter
   - final options watchlist artifact
3. Persist richer watchlist outputs with:
   - watchlist score
   - directional context
   - reason codes
   - options viability score
4. Define the default live operating sequence:
   - build broad watchlist
   - filter for viable options
   - run nightly scan on resulting watchlist
5. Add review outputs for watchlist shrinkage:
   - starting universe size
   - post-underlying-filter size
   - post-options-viability size
   - final scan symbol count

## Exit Criteria
- broad-universe watchlist build is part of the standard daily process
- nightly scan is no longer conceptually tied to a hand-curated 10-symbol loop
- watchlist reduction path is inspectable end-to-end

---

# Phase 4 — Strategy Calibration and Evidence Collection

## Goal
Use the now-stable live data pipeline to evaluate signal sparsity, quality, and regime behavior before enabling stricter live operation.

## Tasks
1. Continue daily paper-live runs over a meaningful observation window
2. Review:
   - pass rate
   - symbol persistence
   - rejection reason distribution
   - directional state distribution
   - strategy-family distribution
3. Track whether:
   - one symbol dominates passes
   - most failures remain `NEUTRAL`
   - bearish names remain misaligned with trend-up regime
4. Only after sufficient data, evaluate:
   - ADX threshold
   - MA stack strictness
   - regime/strategy compatibility rules
   - universe size vs opportunity frequency
5. Document calibration conclusions and keep changes isolated from data-pipeline fixes

## Exit Criteria
- pass frequency is understood, not guessed
- no-trade days are explainable
- any rule changes are evidence-based rather than reactive

---

# Phase 5 — Production Hardening

## Goal
Convert the current paper-live system into a strict-live-capable system with explicit safety controls.

## Tasks
1. Formalize degraded-data policy:
   - no placeholder IV inputs
   - no mock historical fallback
   - no stale breadth inputs
   - no stale option quotes
2. Add breadth freshness checks
3. Add stale/missing data no-trade enforcement
4. Add strict quote-quality policy enforcement
5. Add risk controls:
   - max trades per run
   - max risk per trade
   - max aggregate daily risk
   - optional symbol concentration caps
6. Add runtime monitoring / alerting hooks:
   - degraded mode usage
   - no-candidate days
   - missing provider data
   - quote-quality failures
7. Define a strict-live readiness checklist and make it auditable

## Exit Criteria
- strict-live fails safe on degraded or stale inputs
- risk caps are enforced automatically
- operator can identify why a run was blocked
- strict-live can run unattended without placeholder dependencies

---

# Phase 6 — Paper Execution and Trade Outcome Tracking

## Goal
Upgrade from signal-generation paper-live to true simulated trade lifecycle tracking.

## Tasks
1. Add paper execution path
2. Define fill assumptions:
   - mid
   - conservative midpoint/slippage
   - limit-style assumptions
3. Track:
   - entry timestamp
   - assumed fill price
   - spread width / risk
   - exit rules
   - end-of-day / expiration outcome
4. Produce daily paper execution summaries
5. Add post-trade review outputs

## Exit Criteria
- paper-live includes simulated fills, not just signal generation
- operator can review estimated trade outcomes over time
- strategy can be judged on signal quality and execution realism together

---

# Phase 7 — Strict-Live Readiness Gate

## Goal
Make a formal go/no-go decision for strict-live operation after data quality, calibration, and paper execution criteria are satisfied.

## Tasks
1. Define readiness thresholds:
   - zero placeholder IV inputs
   - acceptable quote-quality metrics
   - acceptable data freshness
   - acceptable paper-live stability window
2. Require passing validation window before enabling strict-live
3. Add a documented operator checklist
4. Optionally support manual approval mode before full unattended runs

## Exit Criteria
- strict-live decision is based on defined readiness criteria
- enabling strict-live is a controlled operational step, not an ad hoc change

---

# Phase 8 — Small-Capital Live Deployment

## Goal
Deploy the strategy live in a constrained, reviewable way.

## Constraints
- small capital only
- strict max-risk limits
- few trades per day
- no degraded-data execution
- optional manual approval at first

## Tasks
1. Add live execution guardrails
2. Add post-trade review artifact
3. Review slippage vs paper assumptions
4. Expand only after stable live behavior is demonstrated

## Exit Criteria
- stable live operations under conservative capital/risk limits
- post-trade review is clean and repeatable
- strategy can be scaled only after evidence supports it
